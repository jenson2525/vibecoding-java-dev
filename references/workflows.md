# 可行性分析维度 与 任务开发流程

本文件供进入编码阶段前阅读：先按「可行性分析」对齐方案，再按任务类型选对应流程。所有流程内部仍严格遵循 SKILL.md 第一节的"单步输出 + 等继续"，并以收尾时的测试验证闸门结束。

## 可行性分析覆盖维度

动手编码前至少覆盖：

- **背景与现象**：要解决什么、当前状态、问题如何表现。
- **方案候选与权衡**：1～N 个可行方案，逐一说明原理、改动范围、优缺点；多方案给出明确推荐及理由。
- **影响面评估**：涉及哪些模块/类/接口，是否影响现有行为、数据结构、对外契约、并发与性能。
- **风险与前置条件**：依赖项、配置、兼容性、回滚成本。

输出后停下等用户拍板。四类流程并非都从第 1 步机械开始——按任务实际进展选起点，但都必须经过可行性分析闸门后才进入编码。

## A. 功能开发（新增能力 / 接入新模块）
1. 需求澄清与可行性分析 — 明确功能边界、输入输出、与现有系统衔接点。
2. 环境配置 — 依赖、配置项、Bean/组件注册、外部资源接入准备。
3. 代码开发 — 自底向上分层：DO/DTO/VO → Repository/Mapper → Service → Controller/接口。每层一步。
4. 测试验证 — 自测方式、调用接口、构造样例、验证预期行为。

## B. 功能优化（重构/性能/结构改善，不改变对外行为）
1. 现状梳理与可行性分析 — 读懂现有实现与调用关系，定位"为什么要优化"，方案**特别强调对现有行为的兼容性**。
2. 改造方案确认 — 明确改造前后差异、保持不变的契约、分几步安全推进。
3. 逐步改造 — 一次只改一个内聚单元，按 code-format.md "修改代码"格式描述每处变更及原因。
4. 回归验证 — 重点验证"行为不变"，必要时补充对比验证。

## C. 问题修复（Bug / 故障排查）
1. 现象与复现 — 收集报错、日志、触发条件，明确复现路径。
2. 定位与根因分析 — 沿调用链/时序排查，区分"表象层"与"根因层"，明确问题出在本方代码、依赖框架还是外部服务。
3. 修复方案确认 — 针对根因给方案（必要时多方案权衡），评估影响面与回归风险。
4. 实施修复 — 最小必要改动，说明"为什么这样改能解决根因"。
5. 验证与防回归 — 确认问题消除，必要时补充防御性校验或日志。

## D. 需求变更（开发中途意图改变）
变更不是另起炉灶，而是让改动有序穿过"分析"这一层：
1. **区分澄清 vs 变更** — 仅是 spec 本就模糊、现在说清楚 → 补全即可；意图真的变了 → 走完整变更流程。
2. **回到可行性分析** — 重新评估方案，重点是**影响面**：哪些已交付代码要返工、哪些接口契约/数据结构要改、回滚成本多大。输出后等用户认可。
3. **逐步返工** — 按 code-format.md "修改代码"格式给出受影响处的改动，逐一说明变更原因。
4. **同步 spec（若已 spec 化）** — 若该特性已沉淀 spec 文档，则更新 spec 与 tasks，并记录"为什么变"；未 spec 化的小功能口头对齐即可。详见 spec-driven.md。
5. **重新验证** — 按测试验证闸门确认变更后行为正确、且未破坏原有功能。
