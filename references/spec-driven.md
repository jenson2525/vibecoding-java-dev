# 轻量 Spec-Driven：何时沉淀 spec、放哪、变更如何穿过它

本范式默认"对话框协作"，不强制写 spec。但对**大特性**，把可行性分析结论沉淀为项目内的 spec 文档，能换来：协作共享、决策留痕、需求演进可回看、AI 上下文精简。原则是**轻量**——只在值得时做，不为每个小功能套模板。

## 何时该 spec 化（判断标准）
满足以下任一条，建议正式 spec 化；否则对话框对齐即可：
- 特性**横跨多次会话 / 多个阶段 / 数周以上**才能完成；
- 需要**多人或多个 agent 协作**同一块；
- 涉及**核心系统、对外契约、数据结构**，未来很可能被回看"当初为什么这么设计"。

> 颗粒度：一份 spec ≈ 一个能独立交付、独立验收的功能单元（约对应一个迭代/一个大 PR）。太大就拆子 spec，太小就并入相关特性。

## 放在哪、拆成几份
进项目 git（便于留痕与协作），按 feature 建目录，每特性一套三件套：

    docs/specs/<feature>/
    ├── spec.md    # 做什么：需求、行为、边界、验收标准（刻意不写实现）
    ├── plan.md    # 怎么做：技术方案、架构决策、组件落点
    └── tasks.md   # 拆成可执行小任务，标状态，每个任务的验收引用 spec 对应条目

全局共享的项目底线（技术栈、风格、架构红线）若需要，单独一份、不随特性拆分。

## spec 与可行性分析的关系
可行性分析就是 spec 的**草稿/前身**：分析里的"背景现象"→ spec；"方案权衡/架构决策"→ plan；"分步推进"→ tasks。spec 化只是把这份分析**正规化、入库、按层级拆开**，不是额外工作。

## 变更如何穿过 spec（配合 workflows.md 的 D 流程）
**改需求 → 先改 spec → 再改代码**，绝不跳过 spec 直接改码、让文档与代码分叉：
1. 在 spec.md 更新"做什么"与验收标准；
2. 评估对 plan.md / 已完成 tasks 的影响，更新方案与任务；
3. 据新 spec 返工实现；
4. spec 是 **living document**：每次变更改文档并（由用户）提交，git 历史即需求演进史。

## 落盘纪律
spec/plan/tasks 同属"过程产物"，遵循 SKILL.md 红线：先在对话框生成供用户审阅，由用户决定是否写入项目，AI 不自动落盘、不做 git 操作。
